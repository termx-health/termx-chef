fsh-chef

REST API for converting between FHIR Shorthand (FSH) and FHIR resources.

Project goal

fsh-chef is a lightweight Express service that wraps the SUSHI and GoFSH compilers and exposes them over HTTP. It compiles FSH source into FHIR JSON resources and vice-versa, so that authoring tools (such as TermX) can offer FSH editing without bundling Node toolchains in the browser.

The V2 API accepts the source content as the raw request body (no JSON envelope) and reads compiler options from the Content-Type header, which makes it convenient to pipe a .fsh or .json file directly from curl or any HTTP client.

V2 API — FSH to FHIR

POST /v2/fsh2fhir accepts the FSH document as the request body. Compiler options ride on the Content-Type header (application/fsh; key=value; ...).

Supported options

  • canonical — canonical URL base
  • version — package version
  • fhirVersion — FHIR version (e.g. 5.0.0)
  • logLevel — silly | debug | verbose | http | info | warn | error | silent
  • snapshot — boolean, include snapshot in output

Example

curl -X POST http://localhost:3000/v2/fsh2fhir \
  -H 'Content-Type: application/fsh; version=0.0.1; fhirVersion=5.0.0' \
  --data-binary $'Profile: MyPatient\nParent: Patient\n* name 1..1'

V2 API — FHIR to FSH

POST /v2/fhir2fsh accepts a FHIR resource (or array of resources) as the request body, with options on the Content-Type header (application/fhir+json; key=value; ...). The response is plain application/fsh.

Supported options

  • logLevel — silly | debug | verbose | http | info | warn | error | silent
  • indent — boolean, indent rules
  • style — map | string

Example

curl -X POST http://localhost:3000/v2/fhir2fsh \
  -H 'Content-Type: application/fhir+json; style=map; indent=true' \
  --data-binary @StructureDefinition-MyPatient.json

Health check

GET /health returns a JSON document with the service status and version, suitable for Kubernetes/Docker health probes. 

$ curl http://localhost:3000/health
{"status":"UP","service":"fsh-chef","version":"1.0.0"}
